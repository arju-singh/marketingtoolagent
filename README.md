# MarketStack — repo + domain → a full marketing suite

Paste a **GitHub repo** and a **project domain**. The tool ingests your real product and
runs **41+ marketing skills** (modeled on
[blurred-machine/marketingskills](https://github.com/blurred-machine/marketingskills)) —
CRO, SEO, copy, emails, ads, launch, pricing, and a 90-day go-to-market plan — each one
grounded in your actual features and copy, generated by **Claude (Mythos 5)**.

## Architecture

A **Vite + React** single-page client and a separate **Node + Express** API. All secrets
(Claude, Razorpay, Supabase service role) live only in the server.

```
client/   Vite + React SPA (React Router)        → the UI
server/   Express API                            → holds all secrets
          /api/ingest  /api/generate[/stream]
          /api/payments/{create-order,verify,webhook}
```

| Layer | Tech |
|---|---|
| Client | Vite + React 19 + React Router (SPA) |
| Styling | Tailwind CSS, Inter via Google Fonts |
| Animation | **GSAP ScrollTrigger + Lenis** (pinned scroll-story), **Framer Motion** reveals, **react-countup**, **CSS keyframe marquee** |
| Server | Node + Express · `@octokit/rest` (repo) · `cheerio` (domain crawl) · Claude TS SDK |
| Data/Auth | Supabase (Postgres + Auth) — optional, app runs without it |
| AI | `claude-mythos-5` → `claude-fable-5` → `claude-opus-4-8` fallback, adaptive thinking, `effort: high`, streaming |

## How it works

```
GitHub URL + domain  ──POST /api/ingest──▶  ProductContext
   (server/src/lib: github.ts + crawl.ts + productContext.ts)
ProductContext  ──POST /api/generate/stream { goals | skills | everything }──▶
   runSkillsStreaming() runs each skill   (server/src/lib: skills.ts + anthropic.ts)
   ──▶ NDJSON stream ──▶ client dashboard of paste-ready Markdown deliverables
```

- **`server/src/lib/skills.ts`** — the full catalog (41+ skills), goal bundles, the "everything" set (shared with the client).
- **`server/src/lib/anthropic.ts`** — the skill runner. Shared product context first (prompt-cache friendly), skill instruction last.
- **`client/src/components/ScrollStory.tsx`** — the pinned GSAP centerpiece.

## Run it

Two processes — the API and the client:

```bash
# install both
npm run install:all

# 1) server  → http://localhost:8787
cp server/.env.example server/.env     # add ANTHROPIC_API_KEY (required for generation)
npm run dev:server

# 2) client  → http://localhost:5173  (proxies /api → the server)
cp client/.env.example client/.env     # add VITE_SUPABASE_* for auth (optional)
npm run dev:client
```

`ANTHROPIC_API_KEY` (in `server/.env`) is the only key needed to generate. `GITHUB_TOKEN`
is optional. Supabase vars are optional — auth/persistence activate only when configured.

### Env

- **`server/.env`** — secrets: `ANTHROPIC_API_KEY`, `ANTHROPIC_MODEL`, `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`, `RAZORPAY_*`, `GITHUB_TOKEN`. See `server/.env.example`.
- **`client/.env`** — public only: `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`, `VITE_API_BASE`. See `client/.env.example`.

## Production setup (physical steps)

Everything runs in **demo mode with zero keys** (simulated payments, client-only gate). To make it real:

**1. Anthropic** — get an API key at console.anthropic.com → set `ANTHROPIC_API_KEY`. Mythos 5 needs Project Glasswing; otherwise it auto-falls back to Fable 5 / Opus 4.8.

**2. Supabase** (auth + history + authoritative paywall)
1. Create a project at supabase.com (Data API **on**, automatic RLS **on**, expose-new-tables **off**).
2. **SQL Editor** → paste & run `supabase/schema.sql` (creates `runs` + `usage` tables, RLS policies, and the `increment_runs` function).
3. **Authentication → Providers → Google** → enable, paste a Google OAuth client ID/secret (from console.cloud.google.com), and add Supabase's callback URL to the Google client's authorized redirect URIs.
4. **Project Settings → API** → **Project URL** → `VITE_SUPABASE_URL` (client) **and** `SUPABASE_URL` (server); **anon public** → `VITE_SUPABASE_ANON_KEY` (client); **service_role** → `SUPABASE_SERVICE_ROLE_KEY` (server-only; enables enforcement).
5. **Authentication → URL Configuration** → add your site + `…/app` to the redirect allow-list.

**3. Razorpay** (payments)
1. Sign up at razorpay.com, complete KYC for live mode (use Test mode first).
2. Settings → **API Keys** → generate → set `RAZORPAY_KEY_ID` + `RAZORPAY_KEY_SECRET`.
3. **Webhook** (robust activation): Settings → **Webhooks** → add `https://yourdomain/api/payments/webhook`, subscribe to **`payment.captured`** (and `order.paid`), set a secret → `RAZORPAY_WEBHOOK_SECRET`. The order carries `{uid, tier}` in its notes, so the webhook activates the plan even if the browser closes mid-redirect.

> **Purchases require sign-in.** When Supabase is configured, the Paywall asks the user to sign in with Google before paying, so every plan is attributed to a user id (the `create-order` route embeds it in the order notes; `verify` and the webhook both activate against it). In demo mode (no Supabase) it unlocks locally.

**4. Deploy (two targets)**
- **Client** (static SPA) → Vercel/Netlify/Cloudflare Pages. Set root dir to `client`, build `npm run build`, output `dist`. Add `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`, and `VITE_API_BASE=https://your-api-host`.
- **Server** (Node) → **Render one-click**: a `render.yaml` Blueprint is included — at render.com → **Blueprints → New**, connect this repo and Render provisions the `server/` web service, then prompts you to fill the `sync: false` secrets (`ANTHROPIC_API_KEY`, `SUPABASE_*`, `RAZORPAY_*`, `GITHUB_TOKEN`, `CLIENT_ORIGIN`). Health check is `/api/health`. (Or any Node host: `npm start`, port from `PORT`.)
- Point the **Razorpay webhook** at `https://your-api-host/api/payments/webhook`.
- Add `https://your-client-host` (and `…/app`) to **Supabase → Authentication → URL Configuration** redirect allow-list.

> Node is pinned to 22 via `.nvmrc`. The server runs `tsx src/index.ts` directly (no build step); the client is a static bundle.

### Verify payments locally

```bash
# 1. start the API with dummy keys so the webhook route is "configured"
RAZORPAY_WEBHOOK_SECRET=testsecret RAZORPAY_KEY_ID=x RAZORPAY_KEY_SECRET=y npm --prefix server start
# 2. in another shell, post a correctly-signed webhook and assert activation
RAZORPAY_WEBHOOK_SECRET=testsecret BASE_URL=http://127.0.0.1:8787 npm run smoke:pay
```

Expected: a valid signature returns `{"ok":true,"activated":"pro"}`; a tampered signature returns `400`.

> With the Supabase `service_role` key set, the paywall is enforced **server-side** in `/api/generate*` (the client gate is just UX). With Razorpay set, the "Pay" buttons open real checkout; a plan activates **only** after `/api/payments/verify` confirms the HMAC signature. Without either, the app still runs in demo mode.

## Freemium & pricing

- **First project analysis is free.** Every run after that requires a plan (`client/src/lib/usage.ts`, `FREE_RUNS = 1`).
- On the 2nd+ run, the **Paywall** (`client/src/components/Paywall.tsx`) offers three one-time tiers: **₹777 Starter · ₹888 Pro (popular) · ₹999 Agency** (`client/src/lib/pricing.ts`).
- Usage is tracked in `localStorage` (optimistic UX) and is authoritative server-side in the Supabase `usage` table when signed in. The configure screen shows free/used/plan status.
- **Payment is simulated** — `Paywall.buy()` has a clear TODO to drop in Razorpay/Stripe checkout + server-side verification. ⚠️ For production, enforce entitlement server-side in `/api/generate` against a real payments record (localStorage is user-clearable).

## Choosing skills & streaming

- **Goals** (default) — pick one or more goals; each runs a curated bundle.
- **Advanced: pick skills** — toggle into an à-la-carte checklist of all 41 skills grouped by category (up to 18 per run).
- **Generate everything** — runs the full non-redundant suite.
- **Streaming** — `/api/generate/stream` emits NDJSON (`start` → one `deliverable` per skill as it finishes → `done`). The dashboard fills in progressively with a live progress bar and per-skill spinners; a single failing skill is reported inline and never aborts the run. The non-streaming `/api/generate` remains for batch/no-UI use.

## Auth, history & export

- **Google sign-in** (`client/src/lib/auth.tsx`) — Supabase OAuth; appears in the nav only when Supabase is configured; the tool works fully signed-out.
- **Run history** — on a successful generate, signed-in users' runs are saved to the Supabase `runs` table (`client/src/lib/runs.ts`), scoped by RLS. Browse and re-download them at **`/runs`**.
- **Export** (`client/src/lib/export.ts`) — every result set exports as a combined `.md` or a `.zip` (one file per deliverable + combined + `product-context.json`). Works with or without sign-in.
- **Security** — Row Level Security (`supabase/schema.sql`): users read/write only their own `runs`; the `usage` table is read-only to clients and written only by the server (service role), so a user can't grant themselves a plan or reset their run count.

## Extending

- Add a skill: append an entry to `SKILLS` in `server/src/lib/skills.ts` (and the shared copy in `client/src/lib/skills.ts`) — key, category, goal, `focus` prompt.
- Persist runs: wire `getAdminDb()` into `/api/generate` to save deliverables per signed-in user.
- The ingest is tolerant: either input can fail and the run still proceeds with whatever it got (`context.notes` records gaps).
