# MarketStack — repo + domain → a full marketing suite

Paste a **GitHub repo** and a **project domain**. The tool ingests your real product and
runs **41+ marketing skills** (modeled on
[blurred-machine/marketingskills](https://github.com/blurred-machine/marketingskills)) —
CRO, SEO, copy, emails, ads, launch, pricing, and a 90-day go-to-market plan — each one
grounded in your actual features and copy, generated by **Claude Opus 4.8**.

## Stack

| Layer | Tech |
|---|---|
| Framework | Next.js 15 (App Router) — React frontend **and** Node backend (route handlers) |
| Styling | Tailwind CSS, Inter via `next/font` |
| Animation | **GSAP ScrollTrigger + Lenis** (pinned scroll-story), **Framer Motion** (`whileInView` reveals), **react-countup** (stats), **CSS keyframe marquee** |
| Backend | Node route handlers · `@octokit/rest` (repo) · `cheerio` (domain crawl) · Claude TS SDK |
| Data/Auth | Supabase (Postgres + Auth via `@supabase/supabase-js`) — optional, app runs without it |
| AI | `claude-opus-4-8`, adaptive thinking, `effort: high`, streaming |

## How it works

```
GitHub URL + domain
   │  POST /api/ingest
   ▼
product-marketing ingest → ProductContext   (lib/github.ts + lib/crawl.ts + lib/productContext.ts)
   │  POST /api/generate { goals | skills | everything }
   ▼
runSkills() runs each chosen skill against ProductContext   (lib/skills.ts + lib/anthropic.ts)
   ▼
Dashboard of paste-ready Markdown deliverables
```

- **`src/lib/skills.ts`** — the full catalog (41+ skills), goal bundles, and the "everything" set.
- **`src/lib/anthropic.ts`** — the skill runner. Shared product context first (prompt-cache friendly), skill instruction last.
- **`src/components/ScrollStory.tsx`** — the pinned GSAP centerpiece.

## Run it

```bash
npm install
cp .env.local.example .env.local   # add ANTHROPIC_API_KEY (required for generation)
npm run dev                         # http://localhost:3000  → /app for the tool
```

`ANTHROPIC_API_KEY` is the only key needed to generate. `GITHUB_TOKEN` is optional
(higher rate limits + private repos). Supabase vars are optional — auth/persistence
activate only when configured.

## Production setup (physical steps)

Everything runs in **demo mode with zero keys** (simulated payments, client-only gate). To make it real:

**1. Anthropic** — get an API key at console.anthropic.com → set `ANTHROPIC_API_KEY`. Mythos 5 needs Project Glasswing; otherwise it auto-falls back to Fable 5 / Opus 4.8.

**2. Supabase** (auth + history + authoritative paywall)
1. Create a project at supabase.com (Data API **on**, automatic RLS **on**, expose-new-tables **off**).
2. **SQL Editor** → paste & run `supabase/schema.sql` (creates `runs` + `usage` tables, RLS policies, and the `increment_runs` function).
3. **Authentication → Providers → Google** → enable, paste a Google OAuth client ID/secret (from console.cloud.google.com), and add Supabase's callback URL to the Google client's authorized redirect URIs.
4. **Project Settings → API** → copy **Project URL** → `NEXT_PUBLIC_SUPABASE_URL`, **anon public** → `NEXT_PUBLIC_SUPABASE_ANON_KEY`, **service_role** → `SUPABASE_SERVICE_ROLE_KEY` (server-only; enables enforcement).
5. **Authentication → URL Configuration** → add your site + `…/app` to the redirect allow-list.

**3. Razorpay** (payments)
1. Sign up at razorpay.com, complete KYC for live mode (use Test mode first).
2. Settings → **API Keys** → generate → set `RAZORPAY_KEY_ID` + `RAZORPAY_KEY_SECRET`.
3. **Webhook** (robust activation): Settings → **Webhooks** → add `https://yourdomain/api/payments/webhook`, subscribe to **`payment.captured`** (and `order.paid`), set a secret → `RAZORPAY_WEBHOOK_SECRET`. The order carries `{uid, tier}` in its notes, so the webhook activates the plan even if the browser closes mid-redirect.

> **Purchases require sign-in.** When Supabase is configured, the Paywall asks the user to sign in with Google before paying, so every plan is attributed to a user id (the `create-order` route embeds it in the order notes; `verify` and the webhook both activate against it). In demo mode (no Supabase) it unlocks locally.

**4. Deploy to Vercel**
1. From the project folder: `npx vercel` (links/creates a project — no git repo required) then `npx vercel --prod` for production. *Or* push to GitHub and "Import Project" at vercel.com — Next.js is auto-detected (build `next build`, no config needed).
2. In **Project → Settings → Environment Variables**, add everything from `.env.local.example` that you've filled in: `ANTHROPIC_API_KEY`, `ANTHROPIC_MODEL`, `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `RAZORPAY_KEY_ID`/`_SECRET`/`_WEBHOOK_SECRET`, and `NEXT_PUBLIC_SITE_URL=https://yourdomain`.
3. Redeploy so the env vars take effect.
4. Point the **Razorpay webhook** at `https://yourdomain/api/payments/webhook` (step 3 above).
5. Add `https://yourdomain` (and `…/app`) to **Supabase → Authentication → URL Configuration** redirect allow-list so Google sign-in works in production.

> Node is pinned to 22 via `.nvmrc`. The OG image runs on the edge runtime; the API routes run on the Node runtime — both are supported on Vercel out of the box.

### Verify payments locally

```bash
# 1. start the server with dummy keys so the webhook route is "configured"
RAZORPAY_WEBHOOK_SECRET=testsecret RAZORPAY_KEY_ID=x RAZORPAY_KEY_SECRET=y npm run start
# 2. in another shell, post a correctly-signed webhook and assert activation
RAZORPAY_WEBHOOK_SECRET=testsecret BASE_URL=http://127.0.0.1:3002 npm run smoke:pay
```

Expected: a valid signature returns `{"ok":true,"activated":"pro"}`; a tampered signature returns `400`.

> With the Supabase `service_role` key set, the paywall is enforced **server-side** in `/api/generate*` (the client gate is just UX). With Razorpay set, the "Pay" buttons open real checkout; a plan activates **only** after `/api/payments/verify` confirms the HMAC signature. Without either, the app still runs in demo mode.

## Freemium & pricing

- **First project analysis is free.** Every run after that requires a plan (`src/lib/usage.ts`, `FREE_RUNS = 1`).
- On the 2nd+ run, the **Paywall** (`src/components/Paywall.tsx`) offers three one-time tiers: **₹777 Starter · ₹888 Pro (popular) · ₹999 Agency** (`src/lib/pricing.ts`).
- Usage is tracked in `localStorage` (optimistic UX) and is authoritative server-side in the Supabase `usage` table when signed in. The configure screen shows free/used/plan status.
- **Payment is simulated** — `Paywall.buy()` has a clear TODO to drop in Razorpay/Stripe checkout + server-side verification. ⚠️ For production, enforce entitlement server-side in `/api/generate` against a real payments record (localStorage is user-clearable).

## Choosing skills & streaming

- **Goals** (default) — pick one or more goals; each runs a curated bundle.
- **Advanced: pick skills** — toggle into an à-la-carte checklist of all 41 skills grouped by category (up to 18 per run).
- **Generate everything** — runs the full non-redundant suite.
- **Streaming** — `/api/generate/stream` emits NDJSON (`start` → one `deliverable` per skill as it finishes → `done`). The dashboard fills in progressively with a live progress bar and per-skill spinners; a single failing skill is reported inline and never aborts the run. The non-streaming `/api/generate` remains for batch/no-UI use.

## Auth, history & export

- **Google sign-in** (`src/lib/auth.tsx`) — Supabase OAuth; appears in the nav only when Supabase is configured; the tool works fully signed-out.
- **Run history** — on a successful generate, signed-in users' runs are saved to the Supabase `runs` table (`src/lib/runs.ts`), scoped by RLS. Browse and re-download them at **`/runs`**.
- **Export** (`src/lib/export.ts`) — every result set exports as a combined `.md` or a `.zip` (one file per deliverable + combined + `product-context.json`). Works with or without sign-in.
- **Security** — Row Level Security (`supabase/schema.sql`): users read/write only their own `runs`; the `usage` table is read-only to clients and written only by the server (service role), so a user can't grant themselves a plan or reset their run count.

## Extending

- Add a skill: append an entry to `SKILLS` in `src/lib/skills.ts` (key, category, goal, `focus` prompt).
- Persist runs: wire `getAdminDb()` into `/api/generate` to save deliverables per signed-in user.
- The ingest is tolerant: either input can fail and the run still proceeds with whatever it got (`context.notes` records gaps).
